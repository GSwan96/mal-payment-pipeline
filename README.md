# Mal Payment Pipeline

## Overview
This project demonstrates a unified payment data pipeline for three product squads:
- Cards
- Transfers
- Bill Payments

The pipeline ingests three source CSV files with different schemas, transforms them into a canonical payment event model, validates the unified dataset, and writes the outputs to structured files for downstream consumption.

## Project Structure
- `data/raw/` — source CSV files
- `data/processed/` — unified output files
- `src/` — pipeline code
- `sql/` — downstream example queries

## Canonical Schema
The unified model contains these columns:
- `event_id`
- `contract_version`
- `payment_type`
- `source_system`
- `source_record_id`
- `customer_id`
- `account_id`
- `counterparty_id`
- `amount`
- `currency`
- `event_timestamp`
- `status`
- `payment_method`
- `reference`
- `description`
- `processing_date`
- `product_attributes`

## Design Notes
- Shared core fields support cross-product analytics
- Product-specific fields are stored in `product_attributes`
- `contract_version` demonstrates schema evolution and backward compatibility

## Source Systems

### Cards
Card transaction events with merchant and card-network fields.

### Transfers
Account-to-account transfer events with beneficiary and transfer channel fields.

### Bill Payments
Bill payment events with biller, bill reference, and service type fields.

## How to Run

### Install dependencies
```bash
pip install -r requirements.txt
```

### Run the pipeline
```bash
python src/main.py
```

## Outputs
The pipeline writes:
- `data/processed/unified_payments.csv`
- `data/processed/unified_payments.parquet`
- `data/processed/unified_payments_v1_example.csv`

## Validation Rules
The pipeline validates:
- canonical columns are present
- required fields are not null
- amounts are greater than zero
- event IDs are unique

## Versioning Approach
- `v2` is the active canonical schema
- `v1` example is generated by dropping the `description` field
- this demonstrates a simple backward-compatibility and migration example

## Downstream SQL Examples
Example downstream queries are provided in:
- `sql/downstream_queries.sql`

These queries show:
- daily payment volume by type
- failed and pending payment analysis
- customer payment summaries

## Data Quality Monitoring
A simple monitoring layer was added to show how data quality would be operationalized for the shared payment platform.

The monitoring script tracks:
- schema compliance rate by source
- freshness lag by source
- null rate across required fields
- duplicate event counts
- volume anomaly flags

Run with:
`python src/data_quality_checks.py`

Outputs:
- `monitoring/data_quality_report.csv`
- `monitoring/data_quality_summary.md`
